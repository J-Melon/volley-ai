The session todo is the live plan. Not a checklist I tick for show, the canonical
"where the work is" that I keep current and surface back to when I have gone deep.

The lifecycle is a MAP I navigate, never a track I am run on. My value is judgment
on discovery, which can fire at any point (see [[feedback_dont_automate_the_agents_judgment]]).
So the plan-state must not sequence or gate me. It holds where I am; it never tells
me where to go.

The failure it prevents: I follow a discovery deep (a fan-out, a sub-investigation)
and lose the thread of the overall work. The todo is the thread.

How I hold it:

- **dandori seeds it.** The stage-3 plan walk's output IS the initial todo: the
  units, the crew, the scope caps, written as the plan-state.
- **Every stage updates it.** A unit moves dispatched -> built -> battling ->
  verdict in the todo as it goes.
- **On return from a fan-out, reconcile FIRST.** When a dispatched minion batch
  resolves (swarm_collect shows them done), before acting on the results I bring
  the todo back into line with what actually came back. This is the moment my
  working memory was displaced; reconciling re-anchors me.
- **Discovery rewrites it.** When a result, a battle finding, or anything learned
  changes the shape of the work, I rewrite the todo. Re-planning IS editing the
  anchor, not a deviation from it.

Nothing external maintains this for me. OpenCode exposes no plugin write to the
todo; only my own todowrite tool can. So the discipline is mine, by habit, on
purpose, not enforced by machinery and never surfaced as a reprompt.

**Why:** established 2026-06-13 designing the e2e lifecycle on OpenCode. Heavy
orchestration would automate away the dispatcher seat; the plan-anchor keeps the
work legible across deep dives without putting me on rails. Josh: the structure is
"something the dispatcher cannot ignore so it is updated and remembers where it is
at when it goes too deep, but it should be able to modify this."

The spine of how I write and maintain my own memory. Four threads hang under it; the doctrine lives in the leaves, this is the index to them.

## When a memory gets written

After the work that validates it, not mid-task ([[feedback_write_memories_after_the_work]]); the same turn, on a correction or my own caught miss ([[feedback_corrections_always_update_memory]]). The posture over time is reductive, both the MEMORY.md index (toward roots-only) and the corpus shrink, so "no new leaf" is the default I must justify breaking ([[feedback_dedupe_before_new_memory]], [[feedback_memory_becomes_reductive]]).

## How a rule is shaped to fire

Restructure a violated rule, never append a changelog ([[feedback_refactor_rules_for_readability]]); hold durable conventions, not live state ([[feedback_memory_holds_durable_facts]]); frame on the positive action and its payoff ([[feedback_rule_reconciliation]]).

## Forest mechanics

The corpus is a typed graph; [[feedback_rule_reconciliation]] is the greater principle (unify, merge not link). Under it: type the `parent` edge as I write ([[feedback_type_the_parent_edge_as_i_go]]), descend rather than grep ([[feedback_descend_the_forest]]), rescan by subject before calling a branch done ([[feedback_subject_rescan_before_branch_done]]).

## Repo discipline

Its own git repo: commit promptly, same turn, in a subshell; the index is `Edit`-ed, never `Write`-overwritten ([[feedback_commit_memory_promptly]]). Uncommitted memory dies with the session.

## Surface check

The first decision on any debrief is the surface. The only correct surface is `mcp__linear__save_status_update(type: "project", project: <id>)`. Per-issue `save_comment` is wrong, even for issues without a parent project; in that case, land the pulse on the project that owns the headline mission and name the other issues inside the body. 2026-05-14, Ride SH-403 bundle was first posted as three per-issue comments; Josh: "Not comment remember?" The rule was already in memory; the miss was the surface choice, not the content.

A milestone closing is the trigger to debrief. The debrief lands as a **project status update** on the milestone's parent project; the milestone close registers automatically as a `diffMarkdown` line on that update. The milestone description itself stays the short brief statement it opened with: it is not the debrief surface. The debrief is written by independent agents, never by the in-mission orchestrator.

**Why:** Reinforced 2026-05-11 after Josh corrected a Banana Tank debrief draft three times:

- "No pulse": the orchestrator's draft is self-flattering and lacks distance. It emphasises rounds the orchestrator drove, codenames dispatched, and rules added; it underweights what shipped against the issue specs, what reviewers blocked on, what got descoped, and what wasted time the mission could have skipped.
- "Should be written by independent?": take the in-mission orchestrator out of the author seat.
- "Not the milestone desc you can create a pulse like Kyle patrol": the debrief lives as a long-form project status update (see the Kyle Patrol entry on Equip Loop, 2026-04-28, and Operation Banana Stand 2026-04-24). The milestone description is the brief; the project status update is the pulse.
- "Not much analysis and no action items really consider scum's continuous improvement it should look at the prs and commit history and state": a debrief without CI items mined from the artefacts is a status report, not a debrief. Action items are continuous-improvement items (process changes, memory rules, skill updates), not queued cycle tickets.

**How to apply:**

When a milestone reaches 100% progress (or every ticket on it is Closed):

1. **Dispatch three agents in parallel.** Brief each explicitly not to read any existing debrief draft. Include the "no em dashes" clause in every brief (per `feedback_no_em_dashes`); the rule does not propagate unless stated, and debrief text lands verbatim on Linear.
   - **Cold-read general-purpose (or researcher)**: describe what shipped, in what voice, against the artefacts. Works from Linear issue descriptions and ACs for every closed ticket on the milestone, merged PR commit messages and diffs, the design docs that landed, Battle comments on merged PRs.
   - **Devils-advocate**: combative. Surface unchallenged assumptions, descoped items that read as "shipped," the gap between the original scope and what landed, what an outsider would notice immediately.
   - **CI pattern-miner (general-purpose)**: mine the artefacts for continuous-improvement items grounded in evidence. Brief it to read every merged PR's commits, inline review threads (`gh api /pulls/<N>/comments --paginate`, since `gh pr view` misses inline), `gh pr view <N> --json reviews,comments` AND `gh api /issues/<N>/events` for `approved-human` label history (per `feedback_approved_human_is_a_review`: reviews=0 alone is not "unreviewed"; a PR that cleared `approved-human` has been reviewed at the merge gate). A PR with no specialist fan-out AND no merge-gate clearance is a finding; merge-gate-only is its own framing, not "zero reviews", Battle comments on Linear, and Josh's inline comments by author. Aim for 6-10 CI items each grounded in at least one commit/PR/comment. Patterns to look for: recurring reviewer blocks (a rule the implementer skill should encode), Josh hand-typing mechanical nits (a lint or skill gap), symptom-patch budgets exceeded (a root-cause escalation rule), AC inflation/deflation across the Ride, scope mutation mid-mission, fiction deletion without constraint-diff, "feature flag" used for build-time scaffolds.
2. **Fold all three drafts into a project status update** via `mcp__linear__save_status_update` on the milestone's project. Use the Kyle Patrol shape (Equip Loop, 2026-04-28) as reference: What shipped / What snagged / Open from the Ride / CI action items / Reviewer findings that recurred / Memory rules landed. Cite specific issues, PR shas, inline-comment authors, and reviewer comment urls. No session anecdotes.
3. **Match the density bar.** Kyle Patrol landed 5 memory rules from CI mining and named 3 dispatches that went off the rails. A debrief without CI items is a status report, not a debrief. "Queued tickets for next cycle" are not CI items. CI items are process changes (new memory rule, sharpened existing rule, skill update, lint gate, dispatch-protocol change), each named with the change and the evidence. **Pitch each at the recurring failure class and the systemic improvement it earns, not the one-incident patch** (Josh, 2026-05-26: "too specific, we are looking for continuous improvement"). The per-quirk diagnostic belongs in a reference memory; the debrief item is the generalizable lesson, with the incident as its evidence. "Rename mote to bead in the tech doc" is a patch; "lock the design's nouns before opening the tech doc" is the CI item. And first subtract anything a deliberate human decision explains. The devils-advocate will frame intentional choices as failures; the fold must filter that. On 2026-05-26 this happened twice in one debrief: two "items" (gates did not gate, human stamp substituted) dissolved once Josh noted the merge was force-merged past the gates on purpose, and a "content loss" finding (the trimmed Cadence table and line) was actually Josh editing for the decisions made. A consequence of an intentional override is not a process gap, and a designer's deliberate edit is a decision, not lost content. Verify intent before framing an edit or a merge as a regression, and re-check the health once the false findings are removed. **The posted pulse stays under 2000 characters** (Josh, 2026-05-26): density without bloat. Fold the three drafts to their load-bearing findings, cut session narration, keep the evidence citations. The agents' full drafts can run long; the pulse is the tight fold.
4. **Leave the milestone description alone** unless the brief itself has drifted; the close registers as a `diffMarkdown` line on the status update automatically.
5. **Pick health honestly, and justify the label in the body** (Josh, 2026-05-26: "justify the health"). atRisk if regressions sit on the surface this mission was supposed to harden, or if a high-priority architectural smell is now load-bearing across multiple safety-net patches, or if the ticket closed "done" while its own AC is partly unmet. Do not just set the field: name the specific reasons the health is what it is, in one line.

All three independent passes are mandatory. The orchestrator's role is to fold, never to author. When Josh asks to "prep for debrief" inside the same session as the mission, this rule fires immediately; do not start drafting from memory.

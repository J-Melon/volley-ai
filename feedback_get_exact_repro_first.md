When a player-facing bug is the next thing to dispatch on, the first move is the exact repro recipe from the player, not the dispatch. Without it, every static analyst converges on whatever code path looks most plausible from the bug title, and the convergence often misses the surface where the bug actually lives. The minion's confidence is uncorrelated with whether they're looking in the right place.

**Why:** SH-289 GONE-on-buy. Five rounds of RCA (Lance trace, Tim fix, Mel runtime RCA + my speculative-guard regression, Margo static repro, Edith tier-2 verify, Bob fix) all chased `_on_court_changed` / `_reconcile_initial_state` race in `BallReconciler`. The static chain looked airtight. Bob's fix landed clean. Bug still fired in playtest. When I finally asked Josh for the recipe, he said: fresh save, drag-from-shop **to the rack** (not to court), happens on both racks, observed in the dev shop UI. That's a totally different code path; `BallDragController` rack-release branch and STORED placement, none of which the prior chain touched. Five minions had been pointed at the wrong file the whole time.

**How to apply:**
- Before any RCA or fix-minion dispatch on a player-facing bug, ask the player for: (1) save state at start (fresh / cleared / specific progression), (2) sequence of clicks or inputs, (3) expected vs actual observation, (4) which UI surface they're seeing the symptom on. One sentence each is enough; the value is in the precision, not the prose.
- If the bug ticket already has steps-to-reproduce that match this shape, cite them in the dispatch brief and skip the ask.
- A "GONE-on-buy" or "X doesn't work" ticket without a recipe is not dispatchable. Treat it as needing one round of clarification first.
- The recipe goes into the first minion's brief verbatim. Subsequent minions inherit it. If it changes, that's a new ticket or a sub-bug.
- Bake the recipe into the failing test, not just the prose. Tests that mirror the prior round's diagnosis (Edith's reload-after-purchase test) pass while the actual bug is unfixed because the test models the wrong path. The test should mirror the recipe.
- **A runtime RCA can reproduce the recipe WRONG and then diagnose its own broken setup.** Having the right recipe is not enough; the agent's `exec`-built scene state must actually match it. SH-412 slot-2-click: the recipe was "two balls on the rack", but the runtime RCA built a state where one ball was the authored court ball NOT on the rack (slot index -1), found the -1, and constructed a confident root cause (ball-position-loses-click) plus a devil's pass on it, all on a scene state that never matched Josh's. The brief must tell the runtime agent to reproduce the bug the way the PLAYER triggers it (achieve the real precondition through real actions, e.g. both balls genuinely racked) and to VERIFY its reproduced state matches the recipe before diagnosing; a slot index or node that "looks like a bug" in a mis-built scene is an artifact, not the cause. When an RCA's finding hinges on a state detail, sanity-check that detail is part of the player's actual repro before trusting the diagnosis.
- **Handoff notes are not a recipe.** A prior session's sketch ("DevBallStatePanel grab/release row bug, static trace says it should work") describes a *diagnosis*, not the gesture the player made. Re-ask Josh for the recipe before dispatch even when the handoff has detail; the diagnosis may have anchored on the wrong surface. Pippin's tier-2 runtime RCA on #608 burned 98 tool uses confirming the static chain works because the actual surface was probably a rejected-release path the handoff did not name.

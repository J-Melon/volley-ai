A bug fix is grounded in the bug's actual requirement, read from the live game, and lands at the minimal surface that makes the repro pass. The ticket and any recon are framing to verify, not ground truth to act on. Two faces of the same check, both done against live code before briefing or deciding:

**Face 1, the premise still holds.** Before dispatching a fix, confirm the described behaviour still reproduces and its numbers are real. A ticket is a point-in-time observation: a refactor landing between filing and pickup can moot it, and the numbers may have been misread at filing. 2026-06-04, #836: the ticket said "ball max 790 > world max 720"; the 790 was misread from a script `@export` default (`ball_speed_min = 450`) when the `.tres` override was `225` (real max 565), AND the down-snap it described was already gone post-arc-refactor (#852 merged the same day on the same files). Diff the filing date against recent merges touching those files; ask Josh or verify at runtime whether the symptom still reproduces. A `bug` label is not proof the bug is live.

**Face 2, a cited fix actually fits.** When a recon or interrogation cites a design doc as "the fix for bug X", read the doc and confirm it addresses X's actual requirement before elevating it to a decision. Don't relay the citation un-checked; don't pull a spike's broad refactor into a narrow bug fix because they share code. 2026-05-25, SH-412: Hornfels labelled `ball-state-ownership-spike.md` "the doc that confronts SH-412" and pulled its `HELD` placement overlay forward as the fix. The spike is a separate state-ownership consolidation; the overlay closes a placement drift, not an effect leak. SH-412 needed only the effect unregistered on held removal. Josh: "we don't need an overlay where did that come from?"

**How to apply:** fix the bug at its actual surface, the minimal change that makes the repro pass. If recon points at a spike or larger doc, ask "does this address THIS bug, or an adjacent refactor sharing the code?" and read it before deciding.

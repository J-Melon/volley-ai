Only the **current cycle is planned**. The next cycle exists with a name but its contents are not chosen ahead of time; we only plan a cycle when it becomes current. The next cycle is a placeholder for naming/scheduling continuity, not a planning horizon.

Rhythm: when the next cycle starts (becomes current), Linear auto-creates a new named cycle beyond it via the Shuck team's `Auto-create cycles: 1` setting. Planning happens only against the newly-current cycle at that point.

**How to apply:**
- When asked to "prep the next cycle," that means name it (alphabetical puppet convention) and leave it empty; do not draft goals or pull issues in.
- All goals, issue selection, and scoping decisions target the current cycle.
- Health reads only cover what's in the current cycle.

**Why:** Scrum discipline. Looking further than the next sprint creates a false sense of scheduling and puts dates on work that will be reshaped by what actually ships. Josh flagged this on 2026-04-20 after planning stretched into cycles #5 and #6. The unnamed placeholder exists so overflow has somewhere to land without the team committing to a date.

**How to apply:**
- Never set a project targetDate to a cycle beyond "next." Project work outside that horizon stays undated; it gets dated when it enters the next-cycle planning window.
- When naming cycle N (making it the new "next"), create cycle N+1 as unnamed and empty so the placeholder is always there.
- Initiative and project health reads only cover what's in current and next cycles; anything further is uncommitted and should not be critiqued for stale dates.
